# DESIGN.md｜心意链接 / xygift UI Lock

## 0. UI Lock 声明

当前 `heartlink-ui-mvp` 中的 UI 是 MVP v0.1 的视觉基准。

后续开发必须以现有 UI 为准，不允许重做、不允许大幅改版、不允许更换视觉风格。本文档不是重新设计 UI 的指导文档，而是现有 UI 的保真规范与视觉保护边界。

如果本文档描述与当前已验收 UI 代码冲突，以当前 `heartlink-ui-mvp` 中的 UI 代码为准。

## 1. UI Source of Truth

视觉源文件 / UI 基准为当前代码中的：

1. `heartlink-ui-mvp/src/app/App.tsx`
2. `heartlink-ui-mvp/src/app/components/CreatorFlow.tsx`
3. `heartlink-ui-mvp/src/app/components/ReceiverFlow.tsx`
4. `heartlink-ui-mvp/src/styles/`
5. `heartlink-ui-mvp/src/app/components/ui/`

其中 `CreatorFlow.tsx` 与 `ReceiverFlow.tsx` 是当前页面视觉、流程布局、文案密度、动效节奏和状态呈现的核心基准。`src/styles/` 提供全局字体、Tailwind 入口和主题 token。`src/app/components/ui/` 保留 shadcn/Radix 风格基础组件能力，但当前主流程主要以内联样式实现。

## 2. 当前设计目标

当前 UI 的设计目标是：

1. 温柔。
2. 私密。
3. 克制。
4. 礼物感。
5. 高级留白。
6. 中文信纸感。
7. 移动端优先。
8. 创建端清晰高效。
9. 接收端沉浸有仪式感。

创建端偏“准备一份心意”：让用户感觉自己是在交出心意材料、把心意包装好，而不是在完成一张工具表单。接收端偏“打开礼物”：不要出现后台感、工具感、调试感或产品管理感。

当前新链路的设计表达：

```text
用户交出心意 -> 系统理解心意 -> 模型加工心意 -> 包装成可发送成品 -> 生成链接发送
```

页面文案应优先使用“准备这份心意”“帮我包装这份心意”“心意已包装好”“预览这份心意”“复制心意链接”等表达，避免把主体验说成“AI 文案生成工具”。

## 3. 已锁定的视觉特征

以下视觉特征来自当前已验收 UI，后续不得为了“规范化”而重写：

1. 奶油色背景：主背景为 `#FAF7F0`。
2. 白色圆角卡片：卡片多为 `#FFFFFF`，圆角通常在 18px、24px、28px 附近。
3. 香槟金细节：主要点缀色为 `#C9A66B`、`#E8C98A`、`#DFB87A`。
4. 深咖色主按钮：主行动按钮为 `#473B35`，文字为白色。
5. 正文深咖色：主要文字为 `#3F342F`。
6. 次级文字灰棕色：常用 `#9B8E86`、`#C5BAB2`。
7. 浅边框：常用 `#EAE2D8`。
8. 柔和阴影：使用低透明度咖色阴影，不做强烈投影。
9. 胶囊按钮：主按钮、次按钮和状态提示多使用 pill 形态。
10. 移动端居中卡片：主容器宽度约 390px，页面在桌面中居中呈现移动端体验。
11. 接收端信纸感排版：衬线标题、英文小标签、引用块、分割线、署名和留白共同构成信纸/收据感。
12. 线性 icon：主要使用 `lucide-react` 图标，如 Mail、Heart、Star、MessageCircle、Zap、Gift、Copy、Check、RefreshCw、Edit3、AlertCircle、WifiOff。
13. 字体组合：`Noto Serif SC` 用于标题/正文信纸感，`Noto Sans SC` 用于表单/按钮/提示，`Lora` 用于英文标签与 quote 气质。

## 4. 不允许重新设计

后续开发明确禁止：

1. 不允许重新设计首页。
2. 不允许重新设计创建流程。
3. 不允许重新设计接收端。
4. 不允许更换主色。
5. 不允许更换按钮风格。
6. 不允许把页面改成 SaaS 风。
7. 不允许把页面改成普通表单工具。
8. 不允许大幅增加装饰。
9. 不允许大量使用 emoji。
10. 不允许增加复杂 H5 编辑器感。
11. 不允许为了接功能而破坏现有排版。
12. 不允许新增登录、支付、模板市场、多人协作、社交广场等入口。
13. 不允许用新设计系统覆盖当前 Figma Make + 正式化小改后的视觉。

## 5. 允许的小改范围

后续开发只允许这些 UI 小改：

1. 修复明显 bug。
2. 修复移动端溢出。
3. 修复文字错别字。
4. 替换 mock 文案为真实数据。
5. 接入真实 loading / error / empty 状态。
6. 保持现有样式前提下增加必要状态。
7. 保持现有布局前提下接入表单状态。
8. 保持现有接收端三状态：封面 / 正文 / 完成。
9. 保持当前配色、圆角、留白、字体和按钮风格前提下，做小范围可访问性或适配修复。

任何超出以上范围的 UI 改动，都必须先更新 `PRD.md` 和本文件，再进入开发任务。

## 6. 创建端 UI 约束

创建端必须保持：

1. 清晰。
2. 高效。
3. 步骤明确。
4. 表单易读。
5. 不过度文艺。
6. 不新增复杂编辑器。
7. 不新增登录 / 支付 / 模板市场 / 多人协作入口。

当前创建流程必须保持为：

首页 → 场景选择 → 准备这份心意 → 包装心意 → 选择风格 → 预览这份心意 → 生成心意链接

创建端允许保留步骤进度，因为创建者需要知道流程位置。当前页面结构包括：

1. 首页：品牌、核心欢迎文案、轻流程“填写 → 生成 → 分享”、主按钮“开始创建心意 →”、次级入口“先看看对方会收到什么”。
2. 场景选择：感谢、祝福、道歉、鼓励等心意场景。
3. 准备这份心意：收集收信人、TA 是你的谁、送出原因、具体小事、表达重点、补充话、想要的感觉和署名。
4. 包装心意：生成中 loading、失败/网络错误状态、可编辑的标题/正文/引用/署名/按钮文案。
5. 风格选择：简约便签、柔粉信笺、雾紫诗笺、日光便签，必须保留小型预览缩略图。`vintage-receipt` 仅保留为兼容历史数据的技术 id，前台不展示收据语义。
6. 预览：展示对方打开链接时看到的封面与正文效果。
7. 成功页：展示心意链接、复制心意链接、预览这份心意、再做一份等 MVP 必要操作。

结果页表达必须是“心意成品”，不是“AI 文案结果”。当前包装形式仍为信封 / 信纸体验，未来可以扩展更多心意成品形式，但当前不新增。

## 7. 接收端 UI 约束

接收端必须保持：

1. 沉浸。
2. 像打开一份小礼物。
3. 不显示原型导航。
4. 不显示“创建端 / 接收端”切换。
5. 不显示“模拟失败”等原型入口。
6. 不显示开发阶段词，例如“原型”“MVP”。
7. 不出现管理后台、状态面板、调试菜单或工具栏。

接收端必须保留三状态：

1. 封面态：收信人、引导文案、场景图标、开启按钮、`xygift · 心意链接` 标识。
2. 正文态：英文标签、标题、正文、引用、署名、日期/场景标识、接收按钮。
3. 接收完成态：正文保留，按钮变为完成态，显示接收时间和珍藏提示。

当前接收端的体验重点是“亲启”与“被好好收藏”，不得改成普通详情页或通知页。

## 8. 成功页 UI 约束

成功页只保留：

1. 专属链接。
2. 复制链接。
3. 打开预览。
4. 再创建一份。
5. 隐私提示文案：`拥有链接的人都可以查看，请只分享给你想分享的人。`

不要加入：

1. 生成图片。
2. 查看接收状态。
3. 社交分享系统。
4. 管理页。
5. 登录提示。
6. 支付入口。
7. 删除、过期、统计或二维码等管理能力入口。

## 9. 状态 UI 约束

以下状态组件必须保留，后续真实业务接入时复用：

1. AI 生成中。
2. AI 生成失败。
3. 网络错误。
4. 输入为空。
5. 复制成功。
6. 复制失败。
7. 接收端加载中。
8. 链接不存在。
9. 链接过期。
10. 接收完成。

可以删除原型演示入口，但不能删除这些状态 UI 本身。真实 AI、真实 token 查询、真实网络请求接入后，应把业务结果映射到当前已有状态，而不是另起一套视觉。

## 10. 文案规则

正式 UI 中禁止出现：

1. `[原型]`
2. `模拟生成失败`
3. `MVP`
4. `测试按钮`
5. `Demo`
6. `Mock`
7. `调试`

除非这些词只出现在代码注释或开发文档中，不出现在用户界面。

用户可见文案要保持：

1. 温柔。
2. 克制。
3. 清楚。
4. 不油腻。
5. 不过度互联网梗。
6. 不过度卖萌。
7. 不把技术实现暴露给接收者。

当前隐私提示固定为：`拥有链接的人都可以查看，请只分享给你想分享的人。`

## 11. 样式保护细则

后续修改样式时必须遵守：

1. 不重写 `src/styles/theme.css` 中的主视觉 token，除非先更新 PRD 与本文件。
2. 不把主按钮从深咖色改为蓝色、紫色、绿色等常见 SaaS 色。
3. 不把背景改成纯白后台页。
4. 不把卡片改成硬边框、强阴影或密集仪表盘样式。
5. 不把接收端改成顶部导航 + 内容区的应用页面。
6. 不把创建端改成复杂工具面板、左侧导航或多栏配置器。
7. 不引入大面积插画、渐变球、强装饰背景或模板市场式卡片瀑布。
8. 不大量使用系统 emoji；需要图标时优先延续 `lucide-react` 线性图标。
9. 表单、按钮、选择卡片、toast、错误卡片应延续当前圆角、边框、留白和色彩。

## 12. 后续开发提醒

任何后续功能开发，例如：

1. 接入真实 AI。
2. 接入 Supabase。
3. 生成真实 token 链接。
4. 保存礼物内容。
5. 查询礼物内容。
6. 记录打开或接收状态。

都必须在不破坏当前 UI 的前提下进行。

如果功能接入需要明显改变 UI，必须先更新 `PRD.md` 和 `DESIGN.md`，再进入 `TODO.md` 或具体开发任务，不能直接改。

## 13. Codex / AI 开发守则

后续 AI / Codex 在本项目中开发功能时，应默认遵守：

1. 先读 `docs/PRD.md`、`docs/DESIGN.md`、`docs/ARCHITECTURE.md`。
2. 再读 `heartlink-ui-mvp/src/app/components/CreatorFlow.tsx` 与 `ReceiverFlow.tsx`。
3. 功能接入优先复用现有页面、字段、状态和视觉结构。
4. 只做与任务直接相关的最小 UI 改动。
5. 不以“优化设计”“重构组件”“统一规范”为理由改变当前已验收视觉。
6. 不生成新的产品范围入口。
7. 不修改原始 Figma Make 目录。

当前 UI 是产品资产，不是临时草图。
