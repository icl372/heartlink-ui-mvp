# Supabase Integration Plan

## Current State

HeartLink currently uses `giftService.ts` as the stable create, read, and accept boundary. Created gifts are stored only in an in-memory map and the same-browser LocalStorage mock store. That is sufficient for preview, refresh, and new-tab QA on one browser, but it cannot support cross-device or recipient sharing.

This document plans the Supabase replacement behind the existing service contracts. It does not create a Supabase project, install a client, add environment files, or change runtime behavior.

## Goals

1. A created gift can be read from `/to/:token` on another browser or device.
2. The existing `Gift`, `GiftRecord`, `CreateGiftInput`, and service signatures remain the UI boundary.
3. Unknown, deleted, and expired tokens continue to map to the existing receiver states.
4. Secrets remain server-only.

## Non-goals

- Login, accounts, ownership dashboards, payment, or public galleries.
- Replacing the current UI, token format, `/to/:token` route, or AI path.
- Treating LocalStorage as a production database.

## Table: `gifts`

Use one table named `gifts`. The current `GiftRecord` type is the field mapping reference.

| Column | Required | Notes |
| --- | --- | --- |
| `id` | yes | UUID primary key generated by the database. |
| `token` | yes | Unique indexed public lookup token. |
| `recipient_name` | yes | Displayed on the receiver cover. |
| `sender_name` | yes | Gift signature source. |
| `occasion` | yes | Existing `GiftOccasion` value. |
| `tone` | yes | Existing `GiftTone` value. |
| `amount_text` | nullable | Display text only, never a payment amount. |
| `original_message` | yes | Creator input used to make the gift. |
| `cover_text` | nullable | Short cover copy. |
| `title` | yes | Generated or edited title. |
| `body` | yes | Generated or edited body. |
| `quote` | yes | Generated or edited quote. |
| `button_text` | yes | Product-safe button copy. |
| `accepted_text` | nullable | Short completion detail. |
| `theme` | yes | Existing persisted `GiftTheme` value. |
| `opened_count` | yes | Non-negative integer, default `0`. |
| `accepted_count` | yes | Non-negative integer, default `0`. |
| `created_at` | yes | Database default timestamp. |
| `updated_at` | yes | Database-managed update timestamp. |
| `accepted_at` | nullable | First successful acceptance timestamp. |
| `expires_at` | nullable | Null means no expiry for the MVP. |
| `is_deleted` | yes | Boolean soft-delete flag, default `false`. |

`status` should remain an application-level derived value instead of a conflicting database column: `is_deleted` maps to not-found, an expired `expires_at` maps to expired, and `accepted_at` maps to accepted. This matches the current `GiftRecord` type and avoids timestamp/status drift.

## Token Strategy

1. Keep the current unpredictable 10-16 character token generator and `/to/:token` URL format.
2. Add a unique constraint and index on `gifts.token`; collision retries belong in the server-side create path.
3. Never derive a token from names, messages, ids, timestamps, or other gift content.
4. `is_deleted = true` returns application-level not-found, so a deleted gift does not reveal its former existence.
5. `expires_at <= current time` returns application-level expired, provided the record is not deleted.
6. A token that matches neither condition returns application-level not-found.

## Anonymous Create Strategy

Recommended MVP architecture:

```text
CreatorFlow -> giftService.createGift() -> owned Vercel Function -> Supabase gifts
```

The creator remains anonymous. The browser sends only the existing `CreateGiftInput` to an owned endpoint. The endpoint validates the payload, creates the token, writes the normalized record, and returns the existing `CreateGiftResult` shape.

Do not allow unrestricted browser-side inserts as the first production implementation. Keep RLS enabled and use a server-side Supabase service role key only inside Vercel Functions. This centralizes validation, token collision handling, abuse controls, and future rate limiting without exposing a privileged key.

## Public Read Strategy

```text
/to/:token -> ReceiverFlow -> giftService.getGiftByToken() -> owned Vercel Function -> Supabase gifts
```

The public read endpoint receives a token and returns only its matching, non-deleted record when it is not expired. It maps the record back to `Gift` before returning it to the browser.

RLS should remain enabled. The initial recommendation is no direct anon table-read policy. The owned endpoint performs the scoped lookup on the server. Possession of a sufficiently random token remains the MVP access mechanism, but the endpoint must not support list, search, or arbitrary record-id reads.

## Receiver State Write-back

TODO-039 should use owned server endpoints for receiver mutations as well.

1. Opening a valid gift may increment `opened_count` as a non-blocking best-effort write. Read success must not be blocked when this analytics update fails.
2. Pressing "收下心意" performs an idempotent accept operation: if `accepted_at` is null, set it to the server timestamp and increment `accepted_count` once. If it is already set, return the existing accepted result without incrementing again.
3. A later read derives `Gift.status` from `accepted_at`, so refresh and another device retain the accepted state.
4. The accept endpoint must only mutate the requested token, must not accept arbitrary columns, and must never expose a service role key.

## LocalStorage to Supabase Migration Boundary

After the three production tasks are complete, lookup priority should be:

1. A real token: read from Supabase through the owned server endpoint.
2. A just-created optimistic gift: keep in memory only until the server create response succeeds; then use the server response as the source of truth.
3. Explicit mock tokens: retain local mock and expired-token branches for QA.
4. Unknown tokens: return not-found.

`heartlink_mock_gifts` remains a same-browser mock preview store only. It must not override a real token response and `MOCK_GIFT` must never be the fallback for arbitrary production tokens.

## Environment Variables and Secrets

No environment file is created by this plan.

| Variable | Visibility | Intended use |
| --- | --- | --- |
| `VITE_SUPABASE_URL` | public, optional | Only if a future browser-side non-privileged Supabase client is explicitly approved. |
| `VITE_SUPABASE_ANON_KEY` | public, optional | Only alongside strict RLS and an explicitly approved browser client. |
| `SUPABASE_SERVICE_ROLE_KEY` | server-only | Vercel Function database access. Never use a `VITE_` prefix. |

For TODO-037 through TODO-039, the recommended path does not require a browser Supabase client. The service role key is configured only in Vercel Project Settings. Never commit `.env` or `.env.local`, and never place real values in documentation, code comments, or chat.

## Error Mapping

| Server outcome | Existing UI outcome |
| --- | --- |
| No matching token or deleted record | receiver `not-found` |
| Matching record with expired `expires_at` | receiver `expired` |
| Server/database network failure during read | do not mislabel as not-found; TODO-038 must preserve loading safety and add only the smallest existing-compatible retry/error treatment if needed |
| Create validation failure | existing creator validation UI where possible |
| Create write/network failure | existing creator error/network boundary, without changing layout |
| Accept write failure | remain on the readable letter state, show an existing-compatible retry/error treatment, and do not optimistically persist accepted state as success |

The current `ReceiverState` has no dedicated network-error variant. TODO-038 and TODO-039 must document the smallest UI-compatible handling before adding one; they must not silently convert a network failure into not-found.

## TODO-037 Boundary: Create Gifts in Supabase

- Change: server-side create endpoint, record mapper, `giftService.createGift()` internal path, and only necessary environment/deployment documentation.
- Do not change: CreatorFlow layout, ReceiverFlow, AI prompt, route format, or token helper contract.
- Requires: Supabase project/table/RLS setup and server-only `SUPABASE_SERVICE_ROLE_KEY` in Vercel.
- Acceptance: anonymous creation returns a token and `/to/:token`; the row has the requested content and theme; current local mock mode remains available for QA.
- Manual verification: Vercel deployment creates a gift and the returned token survives reload.

## TODO-038 Boundary: Read Gifts by Token

- Change: server-side token read endpoint, record-to-`Gift` mapper, and `giftService.getGiftByToken()` internal path.
- Do not change: CreatorFlow, theme rendering contract, token/link format, AI path, or local mock QA tokens.
- Requires: the TODO-037 table plus server-side environment configuration.
- Acceptance: a different browser/device can load the same `/to/:token`, with the same title, copy, theme, and dates; unknown/deleted tokens show not-found; expired tokens show expired.
- Manual verification: test a new browser profile or device, not only a new tab in the creator browser.

## TODO-039 Boundary: Persist Receiver State

- Change: owned open/accept endpoints or one scoped mutation endpoint, `giftService.acceptGift()` internal path, and the minimum receiver state adaptation needed to restore accepted status after reload.
- Do not change: receiver layout, completion copy structure, AI, token/link format, payment, or login.
- Requires: server-side environment configuration and an idempotent database update.
- Acceptance: first accept sets `accepted_at` and increments `accepted_count` once; refresh preserves the received state; failed writes do not falsely show success.
- Manual verification: accept on one browser and reload or open the link on another browser.

## Risks and Open Questions

1. Anonymous create and write endpoints need abuse controls before broad release. Rate limiting and quotas are separate future work.
2. Token possession is the MVP authorization model. Do not add a public list, search endpoint, or predictable token source.
3. Decide the retention and expiry policy before enabling `expires_at` for real gifts.
4. Confirm Vercel Function runtime and Supabase project region before implementation, but do not create either in this planning task.
